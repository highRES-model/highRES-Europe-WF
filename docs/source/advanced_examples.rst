Modelling to generate alternatives (MGA)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Modelling to generate alternatives (MGA) is a technique to explore system designs beyond purely cost-optimal results. The feasible space around the cost-optimum is generally considered to be "flat", meaning that there are many solutions with comparable costs to the cost-optimum, but that may have other desired features. For more details on MGA and different levels of MGA integration into energy systems modelling workflows see `Lombardi et al. (2025) <https://doi.org/10.1016/j.joule.2025.102144>`_.

The MGA workflow implemented with highRES is based on a targeted min/max approach, sometimes called the 'One-At-a-Time' (OAT) approach. The process can be summarised as follows:
1. A baseline model scenario is generated by running the model in its normal mode, minimising the total system cost, resulting in a cost-optimal objective value :math:`f(x^*)`.

2. An allowable cost-increase :math:`\varepsilon` (slack) to the cost-optimal total system cost is introduced, which forms a new constraint, allowing the total system cost of subsequent model runs to satisfy

.. math::

      f(x) = f(x^*) \cdot (1 + \varepsilon). 

3. For each MGA scenario, a new objective is introduced, which could involve minimising or maximising the installed capacity of various electricity generation technologies, for each zone or region individually or aggregated.

For additional information on the OAT approach, see `Neumann and Brown (2021) <https://doi.org/10.1016/j.epsr.2020.106690>`_, `Neumann and Brown (2023) <https://doi.org/10.1016/j.isci.2023.106702>`_ or an implementation with highRES by `Vågerö et al. (2024) <https://doi.org/10.1088/2753-3751/ad8e6a>`_. 

The highRES workflow includes a separate MGA model that can be activated by including ``mga_workflow: True`` in the config. Together with the necessary MGA parameters, this will allow you to run an MGA workflow as described above. 

The MGA switch in the config modifies the default highRES workflow, both when it comes to the GAMS code and the Snakemake rules. In GAMS the switch activates a separate MGA module (``higres_mga.gms``)

In ``highres.gms`` this is controlled by the global parameter mga, which by default is set to "OFF". In an mga workflow the switch will be set to "ON".

.. code-block:: gams

   $setglobal mga "ON"

This then includes the mga module ``highres_mga.gms``. 

.. code-block:: gams

   $ifThen.mga "%mga%" == ON

   $include %codefolderpath%/highres_mga.gms

   eq_obj.. obj =E= mga_objective;

   $else.mga

   eq_obj.. obj =E= costs;

   $endIf.mga

The MGA module in its current state includes two new sets, ``mga_z(z)`` and ``mga_g(g)``. ``mga_z(z)`` is a subset of z (which contains the geographical regions on the zonal level), while ``mga_g(g)`` is a subset of g (which contains the generation technologies). 

.. code-block:: gams

   mga_z(z) = yes$(par_mga_z(z));
   mga_g(g) = yes$(par_mga_g(g));

The zones and technologies included in the subsets are later passed to the MGA objective, and govern what the minimisation or maximisation should run over: 

.. code-block:: gams

   eq_mga_obj..
    mga_objective =E=
        sum( (mga_z(z), mga_g(g)),
              var_exist_pcap_z(z,g)
            + var_new_pcap_z(z,g)
        );

The module also includes the cost constraint:

.. code-block:: gams

   eq_max_costs..
    costs =L= par_optimal_cost * (1+par_slack);

The MGA parameters (``par_optimal_cost``, ``par_slack``, ``par_mga_z(z)`` and ``par_mga_g(g)``)are set in the config file

.. code-block:: yaml

   mga_parameters:
      slacks: [
         0.01,
      ]

      mga_objectives: [
         'minimizing',
         'maximizing',
      ]

      mga_tech_groups:
         windOn: ['Windonshore']
         windOff: ['Windoffshore']
         solar: ['Solar']

      mga_zone_groups:
         germany: ['DE']
         all: ['*']

These are processed and handled in the MGA rules ``workflow/rules/mga.smk``. ``rule set_cost_optimal`` takes the total system cost from the cost-optimal model result file to use for the cost constraint. ``rule create_mga_parameters`` takes the mga parameters specified in the config and processes them for ``rule run_mga``. 

Technologies and zones are given as lists and can contain more than one item. It is therefore possible to minimise or maximise over a list of technologies and/or zones. To avoid having to include all zones manually, the asterisk ``'*'`` can be used to include all zones. 

Lastly, ``rule run_mga`` functions similarly to the cost-optimal model, with the difference coming from loading the MGA module. This results in a set of near-optimal model solutions.
